# 🚀 Java GC (垃圾回收) 面试终极指南 (2026版)

> **核心考点**：内存区域划分、对象存活判断、GC算法原理、主流收集器对比（G1/ZGC）、调优参数与故障排查。
> **适用版本**：JDK 8 ~ JDK 25 (涵盖 LTS 及最新特性)

---

## 1. 核心概念与基础 (必问)

### 1.1 什么是 GC？为什么需要 GC？
*   **定义**：Garbage Collection，自动内存管理机制。
*   **目的**：自动回收堆内存中**不再被使用**的对象，防止内存泄漏（Memory Leak）和内存溢出（OOM）。
*   **回收范围**：主要是 **Java 堆 (Heap)** 和 **方法区 (Method Area/Metaspace)**。栈内存随线程结束自动释放，无需 GC。

### 1.2 如何判断对象已死？(两大算法)
#### A. 引用计数法 (Reference Counting)
*   **原理**：对象被引用一次 +1，失效 -1，为 0 则回收。
*   **缺点**：无法解决 **循环引用** 问题（A 引用 B，B 引用 A，但外部无引用）。
*   **现状**：主流 JVM (HotSpot) **未采用**。

#### B. 可达性分析算法 (Reachability Analysis) ✅ (重点)
*   **原理**：从 **GC Roots** 向下搜索，形成引用链。无法到达的对象判定为可回收。
*   **GC Roots 包括哪些？** (面试背诵点)
    1.  **虚拟机栈**：栈帧中的本地变量表引用的对象（局部变量、方法参数）。
    2.  **方法区**：类静态属性 (`static`) 引用的对象。
    3.  **方法区**：常量池引用的对象（字符串常量）。
    4.  **本地方法栈**：JNI (Native 方法) 引用的对象。
    5.  **其他**：同步锁持有的对象、JVM 内部引用等。

> **💡 追问：被标记为“不可达”的对象会立即被回收吗？**
> *   **不会**。至少经历两次标记过程。
> *   第一次标记后，会筛选出是否需要执行 `finalize()` 方法。
> *   如果对象覆盖了 `finalize()` 且未被调用过，JVM 会将其放入 F-Queue 队列，由低优先级线程执行。
> *   **自我拯救**：在 `finalize()` 中，对象可以重新与 GC Roots 建立联系，从而避免回收（但不推荐使用，不确定性大）。

## 2. 垃圾回收算法 (原理层)

| 算法名称 | 核心思想 | 优点 | 缺点 | 适用场景 |
| :--- | :--- | :--- | :--- :--- |
| **标记 - 清除 (Mark-Sweep)** | 1. 标记所有存活对象<br>2. 清除未标记对象 | 简单，不需要移动对象 | **内存碎片化**严重，分配大对象困难 | 早期 CMS (部分阶段) |
| **标记 - 复制 (Copying)** | 将内存分为两块，每次只用一块。存活对象复制到另一块，清空原块。 | **无内存碎片**，实现简单，效率高 | 内存利用率低 (只有 50%) | **新生代** (Eden + Survivor) |
| **标记 - 整理 (Mark-Compact)** | 1. 标记存活对象<br>2. 将存活对象向一端移动，清理边界外内存 | **无内存碎片**，内存利用率高 | 移动对象成本高，需暂停用户线程 (STW) | **老年代** |
| **分代收集 (Generational)** | 根据对象生命周期不同，将堆分为新生代和老年代，采用不同算法。 | 综合最优，符合对象“朝生夕死”特性 | 实现复杂 | **现代 JVM 默认策略** |

## 3. 主流垃圾收集器 (演进与对比) ⭐⭐⭐

### 3.1 经典收集器 (JDK 8 及以前)
*   **Serial**: 单线程，STW (Stop-The-World)。适合客户端模式，小内存。
*   **Parallel Scavenge**: 多线程，关注 **吞吐量** (CPU 运行代码时间 / 总时间)。适合后台计算任务。
*   **CMS (Concurrent Mark Sweep)**:
    *   **目标**：低延迟。
    *   **特点**：首次实现 **并发标记** 和 **并发清除**。
    *   **缺点**：内存碎片多 (需定期 Full GC 整理)、对 CPU 敏感、浮动垃圾问题。
    *   **现状**：JDK 9 废弃，JDK 14 移除。

### 3.2 现代收集器 (JDK 9+ 主流)
#### 🏆 G1 (Garbage-First) —— JDK 9~20 默认收集器
*   **核心变革**：
    *   **取消物理分代**：堆内存划分为多个大小相等的 **Region**。
    *   **逻辑分代**：每个 Region 动态扮演 Eden, Survivor, Old, Humongous (大对象) 角色。
    *   **可预测停顿**：用户设定 `-XX:MaxGCPauseMillis`，G1 优先回收垃圾最多的 Region (Garbage First)。
*   **过程**：初始标记 (STW) -> 并发标记 -> 最终标记 (STW) -> 筛选回收 (STW, 并行)。
*   **适用场景**：大内存 (6GB+)，对延迟有一定要求，兼顾吞吐量。

#### 🚀 ZGC (Z Garbage Collector) —— JDK 11 引入，JDK 15+ 生产可用，JDK 21+ 默认 (部分发行版)
*   **核心目标**：**超低延迟**。停顿时间不超过 **10ms**，且与堆大小无关 (支持 TB 级堆)。
*   **关键技术**：
    *   **着色指针 (Colored Pointers)**：将对象状态信息存储在对象引用的指针上 (64 位指针的高位)。
    *   **读屏障 (Load Barriers)**：在读取对象时检查指针状态，必要时触发重映射 (Relocation)。
    *   **并发整理**：标记、整理、重映射全部并发进行。
*   **适用场景**：对延迟极度敏感的应用 (高频交易、实时系统)，超大内存。

#### 🦙 Shenandoah
*   **地位**：OpenJDK 项目，RedHat 主导。功能与 ZGC 类似 (低延迟、并发整理)。
*   **区别**：使用 **IHI (Index Heap Indirection)** 转发指针而非着色指针。在某些场景下吞吐略优于 ZGC，但 ZGC 在 HotSpot 中集成度更高。

### 3.3 收集器对比总结表 (面试速查)

| 特性 | CMS | G1 | ZGC / Shenandoah |
| :--- | :--- | :--- | :--- |
| **主要目标** | 低延迟 | 可预测延迟 + 高吞吐 | **超低延迟** (<10ms) |
| **堆内存结构** | 物理分代 (连续) | Region (离散) | Region (离散) |
| **整理方式** | 标记 - 清除 (有碎片) | 标记 - 整理 (部分并发) | **完全并发整理** |
| **STW 时间** | 较短 (但随堆增大而增加) | 可控 (取决于 Region 数量) | **恒定** (与堆大小无关) |
| **最大堆支持** | GB 级别 | 数十 GB | **TB 级别** (16TB+) |
| **JDK 状态** | 已移除 | 默认 (JDK 9-20) | 推荐/默认 (JDK 21+) |

## 4. 高频面试题与回答策略

### Q1: Minor GC, Major GC, Full GC 的区别？
*   **Minor GC (Young GC)**: 回收新生代 (Eden + Survivor)。频率高，速度快。触发条件：Eden 区满。
*   **Major GC**: 回收老年代。通常伴随 Minor GC。
*   **Full GC**: 回收整个堆 (新生代 + 老年代) + 方法区。**速度最慢，STW 时间最长**。
    *   **触发原因**：
        1.  老年代空间不足。
        2.  方法区 (Metaspace) 空间不足。
        3.  `System.gc()` 被显式调用 (不建议)。
        4.  CMS/G1 并发模式失败 (Concurrent Mode Failure)。
        5.  大对象直接进入老年代导致空间不足。

### Q2: 什么时候对象会直接进入老年代？
1.  **大对象**：超过 `-XX:PretenureSizeThreshold` 的对象 (如长字符串、大数组)。
2.  **长期存活**：对象在新生代熬过多次 GC (默认 15 次，`-XX:MaxTenuringThreshold`)。
3.  **动态年龄判断**：Survivor 区中，同一年龄段对象总和大于 Survivor 一半，该年龄段对象提前进入老年代。
4.  **空间分配担保失败**：新生代分配时，Survivor 不够，且老年代无法容纳担保。

### Q3: G1 和 CMS 的区别？(经典题)
*   **内存布局**：CMS 是物理分代 (连续内存)；G1 是 Region (离散内存)。
*   **算法**：CMS 基于标记 - 清除 (有碎片)；G1 整体基于标记 - 整理 (无碎片)。
*   **控制方式**：CMS 无法精确控制停顿；G1 可建立停顿预测模型，用户指定 `MaxGCPauseMillis`。
*   **适用性**：CMS 适合中小堆；G1 适合大堆 (6GB+)。

### Q4: 线上遇到 OOM 怎么排查？(实战题)
1.  **保留现场**：确保启动参数加了 `-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp`。
2.  **分析 Dump 文件**：
    *   工具：MAT (Memory Analyzer Tool), JProfiler, VisualVM。
    *   看什么：**Dominator Tree** (支配树) 找占用最大的对象；**Histogram** 看实例数量；**Leak Suspects** (泄漏嫌疑报告)。
3.  **常见原因**：
    *   内存泄漏：静态集合 (Static Map/List) 无限增长、ThreadLocal 未 remove、未关闭的资源 (IO/DB)。
    *   内存溢出：确实需要更多内存 (调整 `-Xmx`)，或存在大对象/大并发。
4.  **GC 日志分析**：查看 GC 频率和停顿时间。如果 Full GC 频繁且回收效果差，基本确认为内存泄漏或配置不当。

### Q5: 如何选择合适的 GC 收集器？
*   **优先原则**：能用最新的就用最新的 (JDK 21+ 首选 **ZGC**)。
*   **JDK 8 环境**：
    *   注重吞吐量 (后台批处理)：`Parallel Scavenge` (默认)。
    *   注重延迟 (Web 服务)：`CMS` (需手动开启 `-XX:+UseConcMarkSweepGC`)。
*   **JDK 11/17 环境**：
    *   通用推荐：**G1** (默认)。
    *   低延迟需求 (<10ms) 或超大堆：**ZGC** (`-XX:+UseZGC`)。

## 5. 常用调优参数 (Cheat Sheet)

| 参数 | 说明 | 示例 |
| :--- | :--- | :--- |
| `-Xms` / `-Xmx` | 堆初始大小 / 堆最大大小 (建议设为一致，避免震荡) | `-Xms4g -Xmx4g` |
| `-XX:MetaspaceSize` | 元空间初始阈值 | `-XX:MetaspaceSize=256m` |
| `-XX:MaxGCPauseMillis` | G1 最大停顿时间目标 (非绝对) | `-XX:MaxGCPauseMillis=200` |
| `-XX:+UseG1GC` | 强制使用 G1 | (JDK 9+ 默认) |
| `-XX:+UseZGC` | 强制使用 ZGC | (JDK 15+ 可用) |
| `-XX:+HeapDumpOnOutOfMemoryError` | OOM 时自动 dump | 必配 |
| `-Xlog:gc*:file=gc.log:time` | 记录 GC 日志 (JDK 9+) | 用于分析 |
| `-XX:NewRatio` | 老年代与新生代比例 | `-XX:NewRatio=2` (老:新=2:1) |

## 6. 2026 年新趋势 (加分项)

*   **Generational ZGC (分代 ZGC)**:
    *   JDK 21 正式商用。
    *   解决了早期 ZGC 没有分代导致的吞吐量问题。现在 ZGC 也有新生代和老年代之分，性能全面超越 G1。
    *   **面试话术**："在 JDK 21 之后，ZGC 已经引入了分代机制，既保持了亚毫秒级的停顿，又大幅提升了吞吐量，是目前大多数场景下的首选。"
*   **虚拟线程 (Virtual Threads) 与 GC**:
    *   虚拟线程数量巨大 (百万级)，但栈内存极小。
    *   对 GC 的影响：对象创建速率可能更快，要求 GC 具有更高的吞吐量和更低的停顿，进一步利好 ZGC。

## 💡 面试回答模板 (示例)

**面试官**：请介绍一下你了解的 GC 收集器，以及你们项目是怎么选的？

**候选人**：
> "我了解主要的收集器演变过程：从早期的 Serial、Parallel，到低延迟的 CMS，再到现在的 G1 和 ZGC。
>
> 1.  **CMS** 虽然实现了并发，但有内存碎片和浮动垃圾问题，且在 JDK 14 已被移除。
> 2.  **G1** 是目前的主流，它通过 Region 划分和可预测停顿模型，很好地平衡了吞吐量和延迟，适合大堆应用。
> 3.  **ZGC** 是未来的趋势，特别是 JDK 21 引入分代 ZGC 后，它能在 TB 级堆下保持 10ms 以内的停顿，且吞吐量也很高。
>
> **在我们项目中**：
> *   如果是 **JDK 8** 的老项目，我们使用的是 **CMS**，并配合参数调优减少 Full GC。
> *   在新迁移的 **JDK 17/21** 微服务中，我们默认直接使用 **G1** 或 **ZGC** (视延迟敏感度而定)。对于核心交易链路，我们开启了 ZGC，因为它的 STW 时间非常稳定，能有效避免长尾延迟问题。"
