# JVM 运行时数据区域与对象机制详解

> **版本说明**：基于 HotSpot 虚拟机整理  
> **适用场景**：Java 基础复习、面试准备、架构理解

---

## 1. 运行时数据区域 (Runtime Data Areas)

JVM 将内存划分为多个区域，各自承担不同的职责。根据线程共享性可分为两类：

*   **线程共享区**：方法区 (Method Area)、堆 (Heap)
*   **线程隔离区**：虚拟机栈 (VM Stack)、本地方法栈 (Native Method Stack)、程序计数器 (PC Register)

### 1.1 程序计数器 (Program Counter Register)
*   **特点**：线程私有，内存占用极小，是唯一一个在 JVM 规范中没有规定任何 `OutOfMemoryError` 情况的区域。
*   **作用**：
    *   作为当前线程所执行的字节码的**行号指示器**。
    *   多线程环境下，CPU 时间片轮转导致线程切换。每个线程都需要独立的计数器来记录上一条指令执行到的位置，以便切换回来后能继续执行。
    *   若执行的是 Java 方法，计数器记录的是虚拟机字节码指令的地址；若执行的是 Native 方法，计数器值为空 (`Undefined`)。
*   **❓ 疑问解答：程序计数器的数据存放在哪？**
    *   **答案**：它本身就是 JVM 规范定义的一块**独立的微小内存空间**。在物理实现上，它通常对应 CPU 的寄存器（速度最快），或者在 JVM 内部实现为一块极小的、线程私有的内存区域。它不属于堆，也不属于栈，是独立的。

### 1.2 Java 虚拟机栈 (Java VM Stack)
*   **特点**：线程私有，生命周期与线程相同。
*   **作用**：描述 Java 方法执行的内存模型。每个方法被执行时，都会同时创建一个**栈帧 (Stack Frame)** 用于存储局部变量表、操作数栈、动态链接、方法出口等信息。每一个方法从调用直至执行完成的过程，就对应着一个栈帧在虚拟机栈中入栈到出栈的过程。
*   **异常**：
    *   `StackOverflowError`：线程请求的栈深度大于虚拟机所允许的深度（如死递归）。
    *   `OutOfMemoryError`：如果栈可以动态扩展，但扩展时无法申请到足够内存。
*   **栈帧内部结构**：
    1.  **局部变量表 (Local Variable Table)**：
        *   存放编译期可知的各种基本数据类型 (`boolean`, `byte`, `char`, `short`, `int`, `float`, `ref`, `returnAddress`)。
        *   **注意**：`long` 和 `double` 类型占用 **2 个** 变量槽 (Slot)，其余占用 1 个。
        *   引用类型存储的是**对象引用地址**（指针或句柄），不是对象本身。
    2.  **操作数栈 (Operand Stack)**：
        *   后进先出 (LIFO)。
        *   用于保存计算过程中的中间结果，也是字节码指令的工作区。
    3.  **动态链接 (Dynamic Linking)**：
        *   指向运行时常量池中该栈帧所属方法的引用。
        *   支持方法调用过程中的动态绑定（多态）。
    4.  **方法返回地址 (Method Return Address)**：
        *   方法退出后，需要返回到调用者的位置。
        *   退出方式：正常退出（有返回值）或 异常退出（无返回值）。

### 1.3 本地方法栈 (Native Method Stack)
*   **作用**：与虚拟机栈类似，但服务于 **Native 方法**（通常由 C/C++ 编写）。
*   **特点**：HotSpot 虚拟机中，虚拟机栈和本地方法栈合二为一。
*   **异常**：同样会抛出 `StackOverflowError` 和 `OutOfMemoryError`。

### 1.4 Java 堆 (Java Heap)
*   **特点**：线程共享，JVM 管理内存中**最大**的一块区域。
*   **作用**：
    *   存放**几乎全部**的对象实例和数组。
    *   是垃圾收集器 (Garbage Collector, GC) 管理的主要区域，因此也称为 "GC 堆"。
*   **内存溢出**：
    *   当堆中没有内存完成实例分配，且堆无法再扩展时，抛出 `java.lang.OutOfMemoryError: Java heap space`。
*   **弹性扩展**：
    *   堆内存可以固定大小，也可以可扩展（通过 `-Xms` 初始值和 `-Xmx` 最大值控制）。
    *   当使用量超过初始值但未达最大值时，JVM 会自动向操作系统申请扩展内存。

### 1.5 方法区 (Method Area)
*   **特点**：线程共享。
*   **别名**：非堆 (Non-Heap) —— 虽然逻辑上属于堆的一部分，但为了区分 GC 策略，常单独称呼。
*   **作用**：存储已被虚拟机加载的**类信息**、**常量**、**静态变量**、即时编译器编译后的代码缓存等。
*   **演变**：
    *   JDK 1.7 及之前：永久代 (PermGen)。
    *   JDK 1.8 及之后：**元空间 (Metaspace)**，使用本地内存（Native Memory），不再受 JVM 堆大小限制（但受物理内存限制）。
*   **异常**：`java.lang.OutOfMemoryError: Metaspace` (或 `PermGen space`)。

### 1.6 运行时常量池 (Runtime Constant Pool)
*   **位置**：方法区的一部分。
*   **来源**：Class 文件中的常量池表在类加载后存入此处。
*   **内容**：存放编译期生成的各种字面量和符号引用。具有动态性（运行期间也可将新的常量放入池中，如 `String.intern()`）。

### 1.7 直接内存 (Direct Memory)
*   **特点**：**不是** JVM 运行时数据区的一部分，也不是 JVM 规范定义的区域。
*   **作用**：NIO (New Input/Output) 引入，基于通道 (Channel) 与缓冲区 (Buffer) 的 I/O 方式。
*   **优势**：避免了在 Java 堆和 Native 堆之间来回复制数据，提高性能。
*   **风险**：不受 Java 堆大小限制，但受本机总内存限制。若分配过多，可能抛出 `OutOfMemoryError`。

## 2. 虚拟机对象机制

### 2.1 对象的创建过程
指普通 Java 对象（不包含数组和 Class 对象）的创建流程：

1.  **类加载检查**：
    *   JVM 遇到 `new` 指令，首先检查该指令的参数能否在常量池中定位到一个类的符号引用。
    *   检查该类是否已被加载、解析和初始化过。如果没有，必须先执行相应的类加载过程。
2.  **分配内存**：
    *   类加载完成后，JVM 在堆中为新生对象分配一块确定大小的内存。
    *   **分配方式**（取决于堆内存是否规整）：
        *   **指针碰撞 (Bump the Pointer)**：堆内存规整（所有用过的内存在一边，空闲的在另一边，中间有一个指针分界点）。只需将指针向空闲方向移动一段距离即可。
            *   *适用 GC*：Serial, ParNew 等带有压缩整理功能的收集器。
        *   **空闲列表 (Free List)**：堆内存不规整（已使用和空闲内存交错）。JVM 需维护一个列表记录哪些内存块可用，分配时从列表找到足够大的空间，更新列表。
            *   *适用 GC*：CMS 等基于标记 - 清除算法的收集器。
    *   **并发安全问题**：
        *   **CAS + 失败重试**：乐观锁机制，保证更新动作的原子性。
        *   **TLAB (Thread Local Allocation Buffer)**：为每个线程在堆中预先分配一小块内存。线程优先在自己的 TLAB 中分配，只有当 TLAB 用完才需要同步加锁。
        *   *配置*：`-XX:+UseTLAB` (默认开启)。
3.  **初始化零值**：
    *   内存分配完成后，JVM 将分配到的内存空间都初始化为零值（不包括对象头）。
    *   这保证了 Java 代码中可以不初始化就使用字段（获得默认值，如 int=0, boolean=false, ref=null）。
    *   若使用 TLAB，此步骤可在分配 TLAB 时一次性完成。
4.  **设置对象头 (Object Header)**：
    *   设置对象的元数据信息：属于哪个类、类的元数据地址、对象哈希码、GC 分代年龄、锁状态标志、偏向线程 ID 等。
5.  **执行 `<init>` 方法**：
    *   按照程序员的意愿进行初始化（执行构造方法，给字段赋值等）。此时，一个真正的对象才算创建完成。

### 2.2 对象的内存布局
HotSpot 虚拟机中，对象在内存中存储分为 3 部分：

1.  **对象头 (Header)**：
    *   **Mark Word**：存储对象自身的运行时数据，如哈希码 (HashCode)、GC 分代年龄、锁状态标志、线程持有的锁、偏向线程 ID、偏向时间戳等。
    *   **Klass Pointer (类型指针)**：对象指向它的类元数据的指针，虚拟机通过这个指针来确定这个对象是哪个类的实例。（注：数组对象还有一个记录数组长度的数据）。
2.  **实例数据 (Instance Data)**：
    *   程序中定义的各种类型的字段内容（包括父类继承下来的和在子类中定义的）。
    *   **存储顺序**：受虚拟机分配策略参数 (`-XX:FieldsAllocationStyle`) 和源码中定义顺序的影响。
    *   **排列规则**：相同宽度的字段总是被分配到一起；在满足此前提下，父类中定义的字段会出现在子类之前。
3.  **对齐填充 (Padding)**：
    *   没有特定含义，仅起占位符作用。
    *   **原因**：HotSpot 虚拟机的自动内存管理系统要求对象起始地址必须是 **8 字节的整数倍**。对象头正好是 8 的倍数，如果实例数据部分不是 8 的倍数，就需要填充字节补齐。

### 2.3 对象的访问定位
建立堆内存和栈内存的联系：

1.  **句柄访问 (Handle)**：
    *   Java 栈中的 Reference 存储的是**句柄地址**。
    *   堆中划分出一块**句柄池**，句柄中包含两部分信息：对象实例数据地址 + 对象类型数据地址。
    *   **优点**：对象在 GC 整理移动时，只需改变句柄中的实例数据指针，Reference 无需修改，稳定性高。
2.  **直接指针访问 (Direct Pointer)** (HotSpot 默认)：
    *   Java 栈中的 Reference 直接存储**对象地址**。
    *   对象头中包含类型数据指针。
    *   **优点**：速度更快，节省了一次指针定位的时间开销。
    *   **缺点**：对象移动时需要修改所有的 Reference。

### 2.4 对象存活的判断
如何判断一个对象是否可以被回收？

1.  **引用计数法 (Reference Counting)**：
    *   对象被引用一次，计数器+1；引用失效，计数器-1。为 0 时可回收。
    *   **缺点**：难以解决**循环引用**问题（A 引用 B，B 引用 A，但两者都已不再被外部使用）。主流 JVM **未采用**。
2.  **可达性分析算法 (Reachability Analysis)** (主流方案)：
    *   **原理**：选取一组称为 **"GC Roots"** 的对象作为起点，从这些节点开始向下搜索，搜索所走过的路径称为引用链。当一个对象到 GC Roots 没有任何引用链相连时，证明对象是不可用的。
    *   **主要的 GC Roots 包括**：
        1.  **虚拟机栈**：栈帧中的本地变量表引用的对象（局部变量、方法参数）。
        2.  **方法区**：类静态属性引用的对象（`static` 变量）。
        3.  **方法区**：常量引用的对象（字符串常量池）。
        4.  **本地方法栈**：JNI (Native 方法) 引用的对象。
        5.  **其他**：JVM 内部引用、同步锁持有的对象等。
3.  **finalize() 方法 (垂死挣扎)**：
    *   即使对象不可达，也不会立即回收。
    *   若对象覆盖了 `finalize()` 方法且未被调用过，JVM 会将其放入 F-Queue 队列，由低优先级线程执行。
    *   在 `finalize()` 中，对象可以重新与 GC Roots 建立联系（自我拯救），从而避免回收。
    *   **现状**：不推荐使用。执行代价高、不确定性大、无法保证调用顺序。替代方案：`try-finally` 或 `Cleaner`。

### 2.5 对象的四种引用强度
引用强度从强到弱：

| 引用类型 | 特点 | 回收时机 | 典型使用场景 | 代码示例 |
| :--- | :--- | :--- | :--- | :--- |
| **强引用 (Strong)** | 最普遍，只要引用存在，对象绝不回收。 | **永不回收** (除非手动置 null 或超出作用域) | 绝大多数对象 `Object obj = new Object();` | `Object obj = new Object();` |
| **软引用 (Soft)** | 描述还有用但非必须的对象。 | **内存溢出前**。当堆内存不足，即将抛出 OOM 时，会回收软引用对象。 | **缓存**：图片缓存、网页缓存。内存够则保留，不够则释放以防崩溃。 | `SoftReference<Object> sf = new SoftReference<>(obj);` |
| **弱引用 (Weak)** | 非必须对象，强度比软引用更弱。 | **下次 GC 时**。无论内存是否充足，只要发生 GC，就会回收弱引用对象。 | **监听器、映射表**：`WeakHashMap`。防止因忘记移除监听器导致的内存泄漏。 | `WeakReference<Object> wf = new WeakReference<>(obj);` |
| **虚引用 (Phantom)** | 最弱，无法通过虚引用获取对象实例。 | **对象被回收时**。唯一目的就是接收对象被回收的系统通知。 | **堆外内存管理**：配合 `ReferenceQueue` 跟踪对象回收，清理直接内存。 | `PhantomReference<Object> pf = new PhantomReference<>(obj, queue);` |

### 💡 核心总结
*   **内存模型**：分清哪些是线程共享（堆、方法区），哪些是线程私有（栈、PC、本地栈）。
*   **对象创建**：重点理解 TLAB 优化和指针碰撞 vs 空闲列表的区别。
*   **垃圾回收**：掌握“可达性分析”是核心，理解 GC Roots 的组成。
*   **引用类型**：面试高频考点，特别是软引用（防 OOM）和弱引用（防泄漏）的区别。
