# JVM 核心配置参数指南：按重要性排序

JVM 的配置参数多达几百个，但真正决定系统生死、性能和稳定性的核心参数其实只有几十个。本文将它们按**重要性**分为四个梯队，从“必须配置”到“进阶调优”，并附带标准生产环境模板。

---

## 🏆 第一梯队：生死线（必须配置）
**如果不配或配错，应用可能直接起不来，或者运行中频繁崩溃/卡顿。**

| 参数 | 作用 | 为什么重要 | 推荐值/策略 |
| :--- | :--- | :--- | :--- |
| **`-Xms`** | **初始堆大小** (Initial Heap Size) | 防止 JVM 启动后频繁向操作系统申请内存，导致性能抖动。 | **设为与 `-Xmx` 相同** (如 `-Xms4g`) |
| **`-Xmx`** | **最大堆大小** (Max Heap Size) | 限制应用能用的最大内存，防止吃光机器内存被 OOM Killer 杀掉。 | 根据机器内存设定 (如物理内存的 60%-70%) |
| **`-XX:+UseG1GC`**<br>*(或 `-XX:+UseZGC`)* | **指定垃圾收集器** | 决定 GC 算法。默认收集器可能不适合你的场景（如 JDK8 默认是 Parallel，停顿长）。 | JDK8/11/17: `G1GC`<br>JDK21+: `ZGC` (默认) |
| **`-XX:+DisableExplicitGC`** | **禁止代码显式调用 GC** | 防止代码中的 `System.gc()` 导致线上服务突然长时间停顿。 | **生产环境必加** |
| **`-Xlog:gc*`**<br>*(JDK9+)*<br>`-XX:+PrintGCDetails`<br>*(JDK8)* | **打印 GC 日志** | **没有日志就无法调优！** 出问题时这是唯一的“黑匣子”证据。 | 输出到文件，带时间戳和详细统计 |

> **💡 黄金法则**：始终将 `-Xms` 和 `-Xmx` 设为**相同的值**。这能避免 JVM 在运行过程中动态扩容堆内存带来的性能开销和不稳定性。

---

## 🥈 第二梯队：稳定性与可观测性（强烈建议配置）
**配置后能显著提升系统稳定性，并在出问题时快速定位原因。**

| 参数 | 作用 | 为什么重要 | 推荐值/策略 |
| :--- | :--- | :--- | :--- |
| **`-XX:MetaspaceSize`** | **元空间初始大小** | 防止应用启动初期因加载类过多触发频繁的元空间 GC。 | 默认通常够用，微服务可设大点 (如 `256m`) |
| **`-XX:MaxMetaspaceSize`** | **元空间最大大小** | 防止内存泄漏（如动态代理生成太多类）导致元空间无限增长撑爆机器。 | 建议设置上限 (如 `512m` 或 `1g`) |
| **`-XX:+HeapDumpOnOutOfMemoryError`** | **OOM 时自动 dump 堆** | 当发生内存溢出时，自动生成快照文件，用于事后分析是谁占用了内存。 | **必加**，配合下一项使用 |
| **`-XX:HeapDumpPath=/path`** | **指定 Dump 文件路径** | 指定生成的 hprof 文件存哪里，否则可能默认存到奇怪的地方找不到。 | 指向有足够磁盘空间的目录 |
| **`-XX:MaxGCPauseMillis`** | **期望的最大 GC 停顿时间** | (G1/ZGC 专用) 告诉 JVM 你愿意容忍多长的停顿，JVM 会据此调整回收策略。 | G1: `200` (毫秒)<br>ZGC: 通常不需要设，它会自动优化 |
| **`-Xss`** | **线程栈大小** | 每个线程占用的内存。设太小易栈溢出 (StackOverflow)，设太大则限制了能创建的线程数。 | 默认通常 1M。高并发场景可设为 `256k` 或 `512k` 以支持更多线程 |

## 🥉 第三梯队：性能微调（视场景配置）
**在第一、二梯队配置完善且遇到特定瓶颈时，才考虑调整这些。**

| 参数 | 作用 | 适用场景 |
| :--- | :--- | :--- |
| **`-XX:NewRatio`** | **新生代与老年代比例** | 默认是 2 (新生代:老年代=1:2)。如果对象存活率高，可调小新生代；如果对象朝生夕死，可调大新生代。 |
| **`-XX:SurvivorRatio`** | **Eden 与 Survivor 比例** | 调整幸存者区大小。如果对象在 Survivor 间复制频繁，可调整此值。 |
| **`-XX:ParallelGCThreads`** | **GC 并行线程数** | 控制 GC 时使用多少个 CPU 核心。默认由 CPU 核数决定。在容器化环境中可能需要手动限制以防抢占业务线程。 |
| **`-XX:ConcGCThreads`** | **并发 GC 线程数** | (G1/CMS) 控制并发标记阶段的线程数，通常设为并行线程数的 1/4。 |
| **`-XX:InitiatingHeapOccupancyPercent`** | **触发并发标记的堆占比** | (G1 专用) 默认 45%。如果老年代增长快，可调低此值以更早开始标记，避免 Mixed GC 失败退化为 Full GC。 |
| **`-Djava.security.egd=file:/dev/./urandom`** | **随机数生成源** | **解决启动慢问题**。某些 Linux 熵池不足会导致 Tomcat/Spring 启动极慢，此参数可加速随机数生成。 |

---

## 🧪 第四梯队：调试与黑科技（慎用）
**除非你是 JVM 专家或正在排查极度疑难杂症，否则不要动。**

| 参数 | 作用 | 风险 |
| :--- | :--- | :--- |
| **`-XX:+PrintFlagsFinal`** | 打印所有最终生效的参数值 | 无风险，用于确认配置是否生效。 |
| **`-XX:+UnlockDiagnosticVMOptions`** | 解锁诊断级参数 | 解锁后才能使用一些隐藏的危险参数。 |
| **`-XX:G1HeapRegionSize`** | 手动设置 G1 的 Region 大小 | 设错可能导致 G1 效率急剧下降甚至无法启动。 |
| **`-XX:-UseBiasedLocking`** | 关闭偏向锁 | 在高竞争多线程场景下可能提升性能，但在低竞争场景下会降低性能。JDK15 后已废弃。 |
| **`-XX:OnOutOfMemoryError="cmd"`** | OOM 时执行脚本 | 可以配置自动重启或报警脚本，但配置复杂，容易误操作。 |

## 🚀 一个标准的“生产环境”启动模板

假设你有一台 **8GB 内存** 的服务器，运行 **JDK 17**，使用 **G1** 收集器：

```bash
java \
  -Xms4g -Xmx4g                      # [关键] 堆大小固定为 4G \
  -XX:+UseG1GC                       # [关键] 使用 G1 收集器 \
  -XX:+DisableExplicitGC             # [关键] 禁止手动 GC \
  -XX:MaxGCPauseMillis=200           # [重要] 目标停顿 200ms \
  -XX:+HeapDumpOnOutOfMemoryError    # [重要] OOM 自动留证 \
  -XX:HeapDumpPath=/data/logs/dumps  # [重要] 指定 dump 路径 \
  -XX:MaxMetaspaceSize=512m          # [重要] 限制元空间 \
  -Xss256k                           # [重要] 支持更多线程 \
  -Xlog:gc*:file=/data/logs/gc.log:time,uptime,level,tags \
                                     # [关键] 记录详细 GC 日志 \
  -jar my-application.jar
```
