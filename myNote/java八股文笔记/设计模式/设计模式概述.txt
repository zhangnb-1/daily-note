7个设计原则：
    开闭原则：对扩展开放，对修改关闭
    单一职责原则：类的职责尽量单一化，主要从业务逻辑出发进行拆分
    里氏替换：在使用父类的地方，其子类要能做到无缝替换
    依赖倒置：
        抽象实例：
            一层的接口 -> 一层的实现
            一层的实现 调用 二层的接口
            二层的接口 -> 二层的实现
        这样的路线即为依赖倒置
        简单来说，就是要面向接口编程，而不是面向实现编程。
    接口隔离：接口尽量做到粒子化，复杂功能用粒子化接口组合得到
                 注：接口过于粒子化，也会导致接口类过于繁杂，代码过于复杂，一切要适度。就像sql的三范式，适当的数据冗余，可以减少关联表查询
    合成复用：尽量使用组合而不是继承来达到复用的目的
        1.B extends A  ×
        2.B:{
            A a
            ...
            }           √
    迪米特法则：总结一句话，能间接关联的类，就不要直接互相直接关联上
    比如：
        A-B 1对1
        B-C 1对N
        C-D N对N
        应设计为：
        A{B b} B{A a; List<C> cList} C{B b; List<D> dList} D{List<C> cList}
        不应增加直接关联 A-C A-D B-D

结构型:
    个人理解：结构型设计模式，本质都是对逻辑的解耦，从不同业务维度，对复杂逻辑的拆分
        ，使复杂逻辑变成一个个单维度、无耦合的逻辑的组合与叠加

    代理：不同的业务逻辑中，有一些公共逻辑，如事务，日志，异常处理等
        ，这些不属于项目本身的逻辑，而是辅助型代码，且大量业务逻辑中都需要加入这些辅助代码
        比如有业务逻辑 A,B,C
        同时设定事务逻辑为 1，日志逻辑为 2，异常处理逻辑为3
        静态代理：即新建一个类，extend A 同时对方法增加 1，2，3中若干逻辑
        动态代理：将1，2，3这些辅助逻辑本身，创建逻辑增加类，再在运行时，将1，2，3等辅助逻辑 与 业务主逻辑组合生成动态字节码增强类，并创建对象进行处理
            与静态代理的区别：表面上看是没有创建新的类，实际上是静态代理没有对增强逻辑直接封装，并设置缺口进行代理目标的填充
        个人想法：如果业务代码本身也有很多公共逻辑的复用，也可以抽取为代理，然后创建一个空逻辑目标，用代理逻辑叠加组合成为复杂逻辑

    装饰：
        装饰模式和代理模式在代码上很类似，但是在处理的事务类型上有区别.
        代理模式不关心具体业务，是一些和业务耦合很低的如权限，日志，事务之类的公共逻辑抽取
        而装饰模式是对业务逻辑的增强，如我们有A,B,C等主业务，以及主业务附带的a,b,c细分业务
            ，a,b,c可以增强到A,B,C等主业务上，且他们的业务参数可能还会有耦合

    适配：和适配器（Adpter）类似是转接器（Convert）,转接器更像是给实例进行转化，让A转为B,是实例参数的对齐,适配器则把接口对齐，
        Convert: OriginBean{id,name}->MyBean{id,name}
        Adapter: OriginService{originMethod();} -> MyService{myMethod(){originService.originMethod}}

    桥接：
        将功能拆分，用小的service类组合成复杂的service类，用来实现复杂的功能
            ，同时组合是在抽象层组合，底层实现可以做到替换，例如spring的依赖注入，可以选择任意实现类来注入，但参数是实现类的父类接口

    组合：在抽象层定义树状结构，在具体实现中区分不同之类
        在抽象层树状结构中定义子类的共同接口
        比如 省市县，三层行政结构，定义共同的统计人口，统计gdp,执行扶贫 等操作
        还比如 文件系统，定义顶层fileNode类，有双击，右键等功能接口
            在子类dir,file中各自实现，文件的双击是展示子文件，文件的双击是执行文件

    外观-门面模式：帮图里的startReview,springboot里的
        public static void main(String[] args) {
            SpringApplication.run(MyGithubProjectApplication.class, args);
        }

    享元：运用共享技术，有效地支持大量细粒度的对象，目的就是为了省内存
        比如学生list可以直接定义两个student模板，一个是性别男，一个性别女
        具体的学生，再传进来id，年龄，姓名即可

        项目中的应用，比如calculator基类，内在状态有构件类型，工程量类型，以及子类calculator的明确的计算逻辑
                外在状态为 quantitySubitem 对象，即每次任务的单独一个子项的所有构件楼层等数据，作为外在状态



行为型： 策略，模板方法，观察者，迭代器，责任链，备忘录，访问者，中介者，解释器
    如何传递请求 ： 解耦发送者与接收者
        责任链、命令、策略、状态
    怎么通知大家 ：一对多通信
        观察者，中介者
    怎么定义算法股价 ：流程控制
        模板方法模式，访问者模式，解释器模式
    怎么管理状态和遍历：辅助功能
        迭代器模式，备忘录模式

    模板方法：抽取公共逻辑到抽象类的非抽象方法中，非公共逻辑用抽象方法
                ，抽象类的抽象方法的调用放在本类已实现的非抽象方法中  个人常用
    策略模式: 状态机
