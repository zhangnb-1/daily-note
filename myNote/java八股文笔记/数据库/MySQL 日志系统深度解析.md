# MySQL 日志系统深度解析 (Logging System)

日志系统是 MySQL 实现 **持久性 (Durability)**、**原子性 (Atomicity)** 以及 **高可用 (HA)** 的核心基石。理解 Redo Log、Binlog 和 Undo Log 的区别与协作机制，是掌握 MySQL 底层原理的关键。

## 1. 三大核心日志详解

### 1.1 Redo Log (重做日志) —— InnoDB 引擎特有
*   **作用**：保证事务的 **持久性 (Durability)**，实现崩溃恢复 (Crash-safe)。
*   **核心技术**：**WAL (Write-Ahead Logging)**，即“先写日志，再写磁盘”。
*   **写入方式**：**循环写 (Circular Write)**。
    *   文件大小固定（通常由 `innodb_log_file_size` 决定），写满后覆盖旧日志。
    *   分为 `write pos` (当前记录位置) 和 `checkpoint` (当前可覆盖位置)。
    *   如果 `write pos` 追上 `checkpoint`，必须停止更新，强制刷盘（Checkpoint 推进）。
*   **物理日志**：记录的是“在某个数据页上做了什么修改”（例如：在 Page X 的 Offset Y 处写入值 Z）。
*   **为什么需要它？**
    *   随机写磁盘（数据页）速度慢，顺序写日志速度快。
    *   减少磁盘 I/O，提升吞吐量。
    *   即使宕机，重启时通过重放 Redo Log 恢复未刷盘的数据。

### 1.2 Binlog (归档日志) —— Server 层所有引擎共有
*   **作用**：
    1.  **数据归档/恢复**：用于全量备份 + Binlog 增量恢复数据。
    2.  **主从复制 (Replication)**：Master 将 Binlog 发送给 Slave，Slave 重放以实现数据同步。
*   **写入方式**：**追加写 (Append Only)**。
    *   文件写到一定大小后切换新文件，旧文件保留，不会覆盖。
    *   可以通过 `expire_logs_days` 设置过期清理。
*   **逻辑日志**：记录的是 SQL 语句的原始逻辑（例如：`INSERT INTO t VALUES (1, 'a')` 或 行变更的详细前后镜像）。
    *   格式：`STATEMENT` (语句), `ROW` (行，默认推荐), `MIXED`。
*   **特点**：不仅记录修改成功的操作，还记录执行时间、线程 ID 等元数据。

### 1.3 Undo Log (回滚日志) —— InnoDB 引擎特有
*   **作用**：
    1.  **事务回滚**：保证事务的 **原子性 (Atomicity)**。如果事务失败，利用 Undo Log 逆向操作恢复数据。
    2.  **MVCC (多版本并发控制)**：存储数据的历史版本，供 Read View 读取旧版本数据，实现非阻塞读。
*   **写入方式**：随事务修改数据时产生，事务提交后，根据策略逐步清理（Purge）。
*   **逻辑日志**：记录的是数据的反向操作（例如：你做了 `INSERT`，Undo Log 记录 `DELETE`；你做了 `UPDATE` (A->B)，Undo Log 记录 `UPDATE` (B->A)）。
*   **存储位置**：存储在 Undo Tablespace 中（早期在 ibdata1，5.7+ 可独立配置）。

## 2. Redo Log vs Binlog (核心对比)

这是面试中最常考的对比题。

| 特性 | Redo Log | Binlog |
| :--- | :--- | :--- |
| **所属层级** | **InnoDB 存储引擎** 特有 | **MySQL Server 层** 实现，所有引擎都有 |
| **日志类型** | **物理日志** (记录数据页的物理修改) | **逻辑日志** (记录 SQL 语句或行变更逻辑) |
| **写入方式** | **循环写** (空间固定，覆盖旧日志) | **追加写** (文件追加，不覆盖，用于归档) |
| **主要用途** | **崩溃恢复 (Crash-safe)**，保证持久性 | **主从复制**、**数据恢复/归档** |
| **写入时机** | 事务进行中不断写入，提交时刷盘策略可控 | 事务**提交时**一次性写入 |
| **文件大小** | 固定大小 (配置决定) | 无限增长 (按文件大小切分文件) |

> 💡 **一句话总结**：Redo Log 是为了让数据库**坏了能修好**（内部一致性）；Binlog 是为了让数据库**丢了能找回**或者**同步给小弟**（外部一致性与归档）。

## 3. Two-Phase Commit (两阶段提交)

**问题背景**：
既然有了 Redo Log 保证崩溃恢复，为什么还需要 Binlog？反之亦然？
*   如果只写 Redo Log：无法做主从复制，无法用 Binlog 恢复历史数据。
*   如果只写 Binlog：Binlog 是追加写，如果写完 Binlog 还没刷 Redo Log 就宕机了，重启后 InnoDB 引擎认为事务没提交（因为 Redo Log 没记），但 Binlog 里却有记录，导致**主从数据不一致**或**恢复数据错误**。

**解决方案**：MySQL 引入了 **两阶段提交 (2PC)** 来保证 Redo Log 和 Binlog 的逻辑一致性。

### 3.1 两个阶段流程

假设执行 `UPDATE T SET c=c+1 WHERE id=1;`

#### 第一阶段：Prepare (准备阶段)
1.  执行器更新引擎数据（内存中）。
2.  引擎生成 **Redo Log**，并将状态标记为 **prepare**。
3.  **此时事务并未真正提交**，Redo Log 已落盘。

#### 第二阶段：Commit (提交阶段)
1.  执行器生成 **Binlog** 并落盘。
2.  调用引擎的提交接口，将 Redo Log 的状态修改为 **commit**。
3.  **事务正式完成**。

### 3.2 崩溃恢复逻辑 (Crash Recovery)

如果在两阶段中间发生宕机，重启时 MySQL 会扫描 Redo Log：

| 场景 | Redo Log 状态 | Binlog 状态 | 恢复动作 | 原因 |
| :--- | :--- | :--- | :--- | :--- |
| **正常提交** | commit | 有 | 无需特殊处理 | 事务已完成。 |
| **Prepare 前宕机** | 无/不完整 | 无 | **回滚** | 事务根本没开始，直接撤销。 |
| **Prepare 后，Commit 前宕机** | **prepare** | **有** | **提交 (Commit)** | Binlog 已存在，说明上层认为可以提交，为了保持一致性，必须重做提交。 |
| **Prepare 后，Commit 前宕机** | **prepare** | **无** | **回滚 (Rollback)** | Binlog 不存在，说明上层还没来得及记录，为了防止数据不一致，必须回滚。 |

> ⚠️ **关键点**：只有当 Redo Log 是 `prepare` 状态 **且** Binlog 存在时，才会自动提交。这保证了“要么都成功，要么都失败”。

## 4. 日志写入与刷盘策略 (性能调优)

### 4.1 Redo Log 刷盘策略 (`innodb_flush_log_at_trx_commit`)
控制 Redo Log 从 OS Cache 刷到磁盘的频率，直接影响性能和安全性。

| 值 | 含义 | 性能 | 安全性 | 推荐场景 |
| :---: | :--- | :--- | :--- | :--- |
| **0** | 每秒刷一次磁盘 | 最高 | 最低 (宕机丢失 1 秒数据) | 测试环境，对数据丢失不敏感 |
| **1** (默认) | **每次事务提交都刷盘** | 较低 | **最高** (只丢失当前未提交事务) | **生产环境默认**，金融/核心业务 |
| **2** | 每次提交写 OS Cache，每秒刷盘 | 高 | 中 (宕机丢失 1 秒数据，OS 挂掉才丢) | 大多数互联网业务 (性能与安全平衡) |

> 💡 **最佳实践**：绝大多数生产环境建议设置为 **1**，如果追求极致性能且能容忍秒级数据丢失，可设为 **2**。绝对不要设为 0。

### 4.2 Binlog 刷盘策略 (`sync_binlog`)
控制 Binlog 写入磁盘的频率。

| 值 | 含义 | 性能 | 安全性 |
| :---: | :--- | :--- | :--- |
| **0** | 写 OS Cache，由系统决定何时刷盘 | 最高 | 低 (宕机/断电丢失大量) |
| **1** (默认) | **每次事务提交都刷盘** | 低 | **最高** |
| **N** | 每 N 次提交刷一次盘 | 中 | 中 (丢失 N 条事务) |

> 💡 **双 1 设置**：为了保证数据零丢失，生产环境通常设置 `innodb_flush_log_at_trx_commit=1` 和 `sync_binlog=1`，俗称“双 1 设置”。虽然性能有损耗，但数据安全最重要。

## 5. 面试总结模板

如果面试官问：“请讲讲 MySQL 的日志系统以及它们是如何配合工作的。”

**建议回答逻辑：**

1.  **分类介绍**：
    *   **Redo Log**：InnoDB 特有，物理日志，循环写。核心作用是 **WAL** 和 **崩溃恢复**，保证持久性。
    *   **Binlog**：Server 层共有，逻辑日志，追加写。核心作用是 **主从复制** 和 **数据归档**。
    *   **Undo Log**：InnoDB 特有，逻辑反向日志。核心作用是 **事务回滚** 和 **MVCC**。
2.  **核心区别**：重点强调 Redo Log 是循环写的物理日志（为了快和恢复），Binlog 是追加写的逻辑日志（为了归档和复制）。
3.  **一致性保障 (2PC)**：
    *   解释为什么要两阶段提交：防止 Redo 和 Binlog 数据不一致。
    *   描述流程：先写 Redo (prepare) -> 写 Binlog -> 写 Redo (commit)。
    *   恢复逻辑：如果卡在 prepare 但有 Binlog，则提交；如果有 prepare 但无 Binlog，则回滚。
4.  **性能调优**：
    *   提到 `innodb_flush_log_at_trx_commit` 和 `sync_binlog` 参数。
    *   说明生产环境通常采用“双 1”策略保证数据零丢失，或者牺牲 1 秒数据换取高性能（设为 2）。

## 6. 复习自测题

1.  **场景题**：如果 `innodb_flush_log_at_trx_commit` 设为 0，数据库宕机了会丢失多少数据？
    *   *答案*：可能丢失最近 1 秒内已提交的事务数据。
2.  **原理题**：为什么 Redo Log 要设计成循环写，而 Binlog 是追加写？
    *   *答案*：Redo Log 关注近期数据的崩溃恢复，空间有限，循环写避免磁盘爆满；Binlog 需要长期归档用于恢复历史数据和主从同步，必须保留所有历史记录。
3.  **深度题**：在两阶段提交中，如果 Prepare 成功后，Binlog 写入前宕机，重启后事务会怎样？
    *   *答案*：事务会被回滚。因为 Redo Log 是 prepare 状态，但找不到对应的 Binlog，为了保证一致性，InnoDB 选择回滚。
