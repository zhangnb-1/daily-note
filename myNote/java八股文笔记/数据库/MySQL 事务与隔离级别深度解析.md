# MySQL 事务与隔离级别深度解析

本文详细解读数据库事务的核心概念，包括 ACID 特性、四大隔离级别、并发问题定义，以及 MySQL InnoDB 引擎如何通过 **MVCC** 和 **锁机制** 解决这些问题。这是后端开发面试中最高频的考点。

## 1. ACID 特性详解（事务的基石）

事务必须满足以下四个特性，缺一不可。

| 特性 | 英文全称 | 含义 | **MySQL 底层实现机制** | 面试关键词 |
| :--- | :--- | :--- | :--- | :--- |
| **原子性** | Atomicity | 要么全做，要么全不做。不能只做一半。 | **Undo Log (回滚日志)**<br>事务失败时，利用 Undo Log 执行逆向操作，将数据回滚到事务前状态。 | 逻辑日志、回滚、逆向操作 |
| **一致性** | Consistency | 事务前后，数据总量和逻辑规则保持一致（如转账总额不变）。 | **是结果，不是手段**<br>由 A、I、D 共同保证的最终状态，依赖业务约束和数据库约束。 | 业务规则、约束、最终状态 |
| **隔离性** | Isolation | 多个事务并发执行时，互不干扰。 | **锁 (Lock) + MVCC**<br>通过不同的隔离级别控制干扰程度。 | 脏读、幻读、读写分离 |
| **持久性** | Durability | 一旦提交，永久保存，即使断电也不丢失。 | **Redo Log (重做日志) + WAL**<br>先写日志再刷盘（Write-Ahead Logging），崩溃后通过 Redo Log 恢复数据。 | 预写日志、Crash-safe、顺序写 |

> 💡 **记忆口诀**：**U**ndo 保原子，**R**edo 保持久，**锁+MVCC** 保隔离，**一致性**是结果。

## 2. 四大隔离级别与并发问题

隔离级别越高，数据越安全，但并发性能越差。

| 隔离级别 | 缩写 | 脏读 | 不可重复读 | 幻读 | 说明 |
| :--- | :---: | :---: | :---: | :---: | :--- |
| **读未提交** | RU | ✅ | ✅ | ✅ | 几乎不用。能读到别人未提交的数据。 |
| **读已提交** | RC | ❌ | ✅ | ✅ | **Oracle 默认**。<br>只能读到已提交的数据。解决脏读。 |
| **可重复读** | RR | ❌ | ❌ | ❌* | **MySQL 默认**。<br>同一事务内多次读取结果一致。解决脏读、不可重复读。<br>*注：MySQL 通过 MVCC+Next-Key Lock 基本解决了幻读。 |
| **串行化** | Serializable | ❌ | ❌ | ❌ | 效率最低。所有事务排队执行，相当于加全局锁。 |

### 2.1 三大并发问题定义

1.  **脏读 (Dirty Read)**
    *   **定义**：读到了别人**未提交**的修改。
    *   **后果**：如果对方回滚了，你读到的就是无效数据。

2.  **不可重复读 (Non-Repeatable Read)**
    *   **定义**：在同一事务内，**两次读取同一行数据**，结果不一样。
    *   **原因**：别的事务修改并提交了该行数据。
    *   **重点**：侧重于 **“修改” (Update/Delete)**。

3.  **幻读 (Phantom Read)**
    *   **定义**：在同一事务内，**两次查询同一个范围**，第二次查到了第一次没查到的**新行**。
    *   **原因**：别的事务插入并提交了新数据。
    *   **重点**：侧重于 **“新增/删除” (Insert)**。

> ⚠️ **注意**：在 MySQL 的 **RR 级别**下，虽然标准 SQL 定义认为不能解决幻读，但 InnoDB 引擎通过 **MVCC (针对快照读)** 和 **Next-Key Lock (针对当前读)** 的组合拳，**实际上已经解决了绝大部分幻读问题**。

## 3. MVCC (多版本并发控制) —— 核心机制

**MVCC (Multi-Version Concurrency Control)** 是一种读写冲突优化机制。
*   **传统锁机制**：读也加锁，写也加锁 → 读写互斥，性能低。
*   **MVCC 机制**：**读不加锁，写不加锁**（读写并行）。读的是数据的“历史版本”，写的是数据的“新版本”。

**主要作用**：
1.  在 **RC 级别** 下解决不可重复读。
2.  在 **RR 级别** 下解决不可重复读 和 幻读（快照读场景）。

### 3.1 MVCC 的三大核心组件

#### 1. 隐藏字段 (Hidden Columns)
InnoDB 在每行记录后面隐藏了三个字段（对用户不可见）：
*   `DB_TRX_ID`：最近修改该行数据的**事务 ID**。
*   `DB_ROLL_PTR`：**回滚指针**，指向该行数据的上一个版本在 **Undo Log** 中的地址。
*   `DB_ROW_ID`：隐藏的行 ID（如果没有主键/唯一索引时使用）。

#### 2. Undo Log (版本链)
*   **作用**：记录数据的历史版本。
*   **工作流程**：
    1.  修改数据时，不直接覆盖，而是将旧数据拷贝到 **Undo Log**。
    2.  更新当前行的 `DB_TRX_ID` 为当前事务 ID。
    3.  更新 `DB_ROLL_PTR` 指向新生成的 Undo Log 记录。
*   **结构**：多次修改后形成一条**链表**（版本链）。

```text
[当前行数据] (trx_id=100, 值="C") 
   ⬇️ (roll_ptr 指向)
[Undo Log 版本 1] (trx_id=80, 值="B")
   ⬇️ (roll_ptr 指向)
[Undo Log 版本 2] (trx_id=50, 值="A")
