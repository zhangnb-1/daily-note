# MySQL 锁机制深度解析 (Locking Mechanism)

锁是数据库解决并发竞争、保证数据一致性的核心手段。InnoDB 引擎的锁机制非常复杂且精细，理解它对于解决死锁、性能优化至关重要。

## 1. 锁的粒度分类 (Granularity)

根据锁定的范围大小，MySQL 锁分为三类：

| 锁粒度 | 名称 | 描述 | 特点 | 典型场景 |
| :--- | :--- | :--- | :--- | :--- |
| **全局锁** | Global Lock | 锁定整个数据库实例。 | **阻塞所有写操作**（DML/DDL），读操作通常也被阻塞（取决于命令）。 | 全库逻辑备份 (`mysqldump`)。 |
| **表级锁** | Table Lock | 锁定整张表。 | 开销小，加锁快；**无死锁**；并发度最低。 | MyISAM 引擎默认；InnoDB 的元数据锁 (MDL)、意向锁。 |
| **行级锁** | Row Lock | 锁定具体的某一行记录。 | 开销大，加锁慢；**可能出现死锁**；并发度最高。 | **InnoDB 引擎的核心**，高并发业务场景。 |

> ⚠️ **注意**：InnoDB **不支持** 页级锁 (Page Lock)。

### 1.1 全局锁 (Global Lock)
*   **命令**：`FLUSH TABLES WITH READ LOCK` (FTWRL)。
*   **作用**：让整个库处于只读状态。
*   **风险**：如果在主库执行，主库无法写入，主从复制会中断。**现代备份通常使用 `--single-transaction` (基于 MVCC) 代替全局锁**。

### 1.2 表级锁的特殊类型
在 InnoDB 中，表级锁主要用于辅助行锁或保护元数据：
1.  **元数据锁 (MDL, Metadata Lock)**：
    *   **作用**：防止在查询过程中表结构被修改（如 `DROP TABLE`, `ALTER TABLE`）。
    *   **机制**：自动加锁，无需手动干预。读操作加 MDL 读锁，写操作加 MDL 写锁。
    *   **坑点**：长事务未提交会持有 MDL 锁，导致后续对该表的 DDL 操作无限等待（甚至拖垮整个实例）。
2.  **意向锁 (Intention Lock)**：见下文详解。
3.  **自增锁 (AUTO-INC Lock)**：插入时保证自增 ID 连续性。

## 2. 行锁的类型：共享锁 vs 排他锁

这是行锁最基本的两种模式，决定了事务间的兼容性。

| 锁类型 | 英文 | 别名 | 作用 | 兼容性矩阵 |
| :--- | :--- | :--- | :--- | :--- |
| **共享锁** | **S Lock** | 读锁 | 允许事务读取一行数据，**禁止修改**。 | S 与 S **兼容**<br>S 与 X **冲突** |
| **排他锁** | **X Lock** | 写锁 | 允许事务读取并修改一行数据，**禁止其他任何锁**。 | X 与 S **冲突**<br>X 与 X **冲突** |

*   **口诀**：**读读兼容，读写互斥，写写互斥**。
*   **SQL 对应**：
    *   `SELECT ... LOCK IN SHARE MODE` (或 `FOR SHARE`) -> 申请 S 锁。
    *   `SELECT ... FOR UPDATE`, `UPDATE`, `DELETE`, `INSERT` -> 申请 X 锁。

## 3. 意向锁 (Intention Locks) —— 表级锁的“信号灯”

**为什么需要意向锁？**
如果没有意向锁，当一个事务想给**整张表**加锁（虽然 InnoDB 很少这么做，但逻辑上需要判断）时，它必须遍历表中每一行，检查是否有行锁。这在大数据量下是不可接受的。

**意向锁的作用**：
它是一种**表级锁**，用于告诉其他事务：“**我这个事务打算在某几行上加锁了，你们别想给整张表加锁！**”

### 3.1 两种意向锁
1.  **意向共享锁 (IS, Intention Shared)**：
    *   事务打算在某行加 **S 锁** 前，先在表上加 IS 锁。
    *   例：`SELECT ... LOCK IN SHARE MODE`。
2.  **意向排他锁 (IX, Intention Exclusive)**：
    *   事务打算在某行加 **X 锁** 前，先在表上加 IX 锁。
    *   例：`SELECT ... FOR UPDATE`, `UPDATE`。

### 3.2 兼容性矩阵 (关键)
意向锁之间、意向锁与表锁之间的兼容性：

| 请求锁 \ 已有锁 | IS | IX | 表 S 锁 | 表 X 锁 |
| :--- | :---: | :---: | :---: | :---: |
| **IS** | ✅ | ✅ | ✅ | ❌ |
| **IX** | ✅ | ✅ | ❌ | ❌ |
| **表 S 锁** | ✅ | ❌ | ✅ | ❌ |
| **表 X 锁** | ❌ | ❌ | ❌ | ❌ |

> 💡 **核心逻辑**：
> *   如果表上有 **IX 锁**（有人要改行），则**不能**加 **表 S 锁** 或 **表 X 锁**。
> *   意向锁和意向锁之间永远兼容（大家只是想改行，互不干扰）。
> *   **意向锁不会阻塞行锁**，它只阻塞表级锁。

## 4. 间隙锁 (Gap Lock) 与 临键锁 (Next-Key Lock)

这是 InnoDB 解决 **幻读** 问题的杀手锏，仅在 **RR (可重复读)** 隔离级别下生效（RC 级别通常不加间隙锁，除非开启特殊配置）。

### 4.1 三种行锁细化类型

1.  **记录锁 (Record Lock)**：
    *   锁定**具体的索引记录**。
    *   例：`id=10` 的记录。

2.  **间隙锁 (Gap Lock)**：
    *   锁定**索引记录之间的间隙**，或者第一条记录之前、最后一条记录之后的范围。
    *   **不包含**记录本身。
    *   **目的**：禁止其他事务在间隙中 **INSERT** 新数据。
    *   例：锁定 `(5, 10)` 开区间，其他事务不能插入 `6, 7, 8, 9`。

3.  **临键锁 (Next-Key Lock)**：
    *   **Record Lock + Gap Lock** 的组合。
    *   锁定一个**左开右闭**的区间 `(prev, current]`。
    *   **默认行为**：InnoDB 在 RR 级别下，对非唯一索引的范围查询，默认加 Next-Key Lock。

### 4.2 加锁规则示例
假设表中有数据：`id = [2, 5, 10, 15]`

| SQL 语句 (RR 级别) | 加锁范围 | 说明 |
| :--- | :--- | :--- |
| `SELECT * FROM t WHERE id = 5 FOR UPDATE` | **(2, 5], (5, 10]** | 唯一索引等值查询，退化为 Record Lock (仅锁 5)。<br>*(注：如果是非唯一索引，会加 Next-Key)* |
| `SELECT * FROM t WHERE id > 5 AND id < 10 FOR UPDATE` | **(5, 10)** | 范围查询，加 Gap Lock。 |
| `SELECT * FROM t WHERE id >= 5 AND id <= 10 FOR UPDATE` | **(2, 5], (5, 10], (10, 15]** | 范围查询，加 Next-Key Lock。锁定 5, 10 以及它们前后的间隙。 |
| `SELECT * FROM t WHERE id = 20 FOR UPDATE` (20 不存在) | **(15, +∞)** | 查询不存在的记录，加 Gap Lock 防止插入。 |

> ⚠️ **重要前提**：锁是加在 **索引** 上的！如果查询条件没有走索引，InnoDB 会锁住 **全表所有记录**（聚簇索引），退化为表锁，性能极差。

## 5. 死锁 (Deadlock) 案例分析与排查

### 5.1 什么是死锁？
两个或多个事务互相持有对方需要的锁，并互相等待对方释放，形成循环依赖。

### 5.2 经典死锁场景

#### 场景 A：交叉更新 (AB-BA)
最常见的情景。
*   **事务 A**：持有 `id=1` 的锁，请求 `id=2` 的锁。
*   **事务 B**：持有 `id=2` 的锁，请求 `id=1` 的锁。
*   **结果**：死锁。InnoDB 会回滚其中一个事务（通常是代价较小的那个）。

#### 场景 B：间隙锁冲突 (Insert + Gap)
*   **事务 A**：执行 `SELECT ... WHERE id = 10 FOR UPDATE`。由于 10 不存在，加了 Gap Lock `(5, 15)`。
*   **事务 B**：执行 `INSERT INTO t VALUES (10)`。需要在这个间隙插入，被 A 的 Gap Lock 阻塞。
*   **事务 A**：又执行 `INSERT INTO t VALUES (10)`。
    *   此时 A 已经持有 Gap Lock，但 Insert 需要申请 **Insert Intention Lock** (一种特殊的间隙锁，与其他 Gap 兼容，但与具体的行锁冲突) 或者直接升级锁。
    *   如果 B 也在等待 A 释放 Gap 锁以便插入，而 A 又在等待 B (比如 B 持有了某种隐式锁)，可能形成死锁。
    *   *更常见的情况*：两个事务都对同一个 Gap 加锁，然后都试图在 Gap 中间 Insert，InnoDB 可能会检测到死锁。

#### 场景 C：唯一键冲突 (Unique Key Conflict)
*   **事务 A**：`INSERT` 一条数据，持有该行隐式锁。
*   **事务 B**：`INSERT` 同一条主键数据，发现冲突，申请 **S 锁** 去检查唯一性，被 A 的隐式锁阻塞。
*   **事务 A**：又去访问另一行，被 B 持有的锁阻塞。
*   **结果**：死锁。
